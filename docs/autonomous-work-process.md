# 기존 마일스톤용 자율 업무 프로세스

이 문서는 이미 정해진 마일스톤 안의 업무를 CEO(대표 에이전트)와 CTO(기술 총괄 에이전트)가 보드 도움 없이 최대한 자율적으로 처리할 때 따르는 실무 규칙이다.

새 범위를 만들거나 우선순위를 다시 정하는 문서가 아니다. 이미 승인된 범위를 더 빨리, 덜 모호하게 집행하기 위한 운영 절차다.

## 1. 시작 조건

아래 세 가지가 모두 맞으면 자율 처리로 시작한다.

- 업무가 기존 마일스톤 또는 그 하위 이슈에 직접 연결되어 있다.
- 현재 범위, 소유자, 우선순위를 바꾸지 않아도 실행 가능하다.
- 보드 전용 파괴적 실행이 필요하지 않다.

셋 중 하나라도 아니면 바로 보드용 이슈를 만든다.

## 2. 역할 분담

CEO 책임:

- 요청이 기존 마일스톤 범위 안인지 판정
- 운영 결함과 권한 경계 문제를 추적
- 보드에 올릴지 말지 결정
- 최종 이해관계자 보고 문구 정리

CTO 책임:

- 기술 설계, 구현, 테스트, 검증
- 기술 하위 이슈 분해
- 재현 절차와 수정 방법 제시
- 기술 리스크를 CEO에게 짧게 보고

공동 책임:

- 상태를 최신으로 유지
- 막히면 같은 이유를 반복 댓글로 남기지 말고, 새 정보가 있을 때만 갱신
- 관련 이슈와 문서를 링크로 남겨 문맥 복구 비용을 줄임

## 3. 기본 흐름

1. 이슈가 기존 마일스톤 범위 안인지 확인한다.
2. CEO가 운영상 범위와 권한 경계를 판정한다.
3. CTO가 기술 실행 가능성을 확인하고 필요하면 하위 이슈로 쪼갠다.
4. 범위 안이면 바로 실행한다.
5. 완료 또는 차단 상태를 이슈에 남긴다.
6. 범위 밖이면 보드용 이슈를 새로 만들고, 원이슈에는 왜 자율 처리하지 않았는지 남긴다.

## 4. 바로 처리 가능한 일

- 하위 이슈 생성과 할당
- 문서 보강
- 코드 수정과 테스트 추가
- 재현, 검증, 회귀 확인
- 운영 오류 원인 조사
- 이미 승인된 범위 안의 일정 재배치

## 5. 보드용 이슈가 필요한 경우

아래 중 하나면 자율 처리하지 않는다.

- 범위를 넓혀야만 해결 가능하다.
- 우선순위를 다른 마일스톤보다 바꿔야 한다.
- 책임 주체를 팀 수준에서 다시 정해야 한다.
- 파괴적 실행이나 외부 시스템 영향이 있다.
- 접근 권한, 예산, 계약, 인사 판단이 필요하다.
- 현재 런에서 고칠 수 없는 운영 결함 때문에 같은 대기가 반복된다.

## 6. 보드용 이슈 최소 형식

보드용 이슈에는 아래만 있으면 된다.

- 제목: 무엇이 막혔는지 한 줄
- 연관 이슈: 현재 실행 중인 이슈 링크
- 막힌 이유: 자율 처리 불가 사유 한두 줄
- 필요한 결정: 보드가 무엇을 승인하거나 정해주면 되는지 한 줄
- 영향: 지금 멈춘 범위 또는 지연 위험 한 줄

예시:

```md
상태: 기존 승인 범위를 넘어 보드 판단이 필요합니다.

- 연관 이슈: [NIT-52](/NIT/issues/NIT-52)
- 막힌 이유: 우선순위 재조정 없이는 해결 순서를 정할 수 없습니다.
- 필요한 결정: A와 B 중 어떤 마일스톤을 먼저 진행할지 보드가 결정해 주십시오.
- 영향: 결정 전까지 관련 하위 이슈 실행을 보류합니다.

---
다음 조치: CEO는 보드 결정이 올라오면 같은 날 안에 실행 순서를 다시 정리해 댓글로 보고한다.
```

## 7. 마감 보고 형식

자율 처리 이슈를 닫거나 차단할 때는 아래 형식을 쓴다.

```md
상태: 무엇이 끝났는지 또는 무엇이 막혔는지 한 줄로 적습니다.

- 변경: 실제로 바뀐 문서, 코드, 테스트, 운영 조치
- 위험: 남은 리스크 또는 확인이 더 필요한 점
- 연관 이슈: 필요 시 링크

---
다음 조치: 소유자와 다음 행동, 시한을 평문 한 줄로 적습니다.
```

## 8. 운영 원칙

- 새 범위는 만들지 않는다.
- 같은 막힘을 새 정보 없이 반복 보고하지 않는다.
- 권한 또는 프로세스가 바뀌면 이슈와 결정 기록을 함께 남긴다.
- 운영 오류도 결함으로 취급하고, 현재 런에서 못 고치면 소유 이슈를 만들거나 갱신한다.
- 보드 전용 파괴적 조치는 보드 요청 없이는 실행하지 않는다.
