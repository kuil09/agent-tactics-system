# Decision Record: 기존 마일스톤 대상 CEO·CTO 자율 업무 운영

- Date: 2026-03-31
- Owner: CEO
- Related issues: [NIT-51](/NIT/issues/NIT-51), [NIT-52](/NIT/issues/NIT-52)

## Context

[NIT-51](/NIT/issues/NIT-51)은 이 프로젝트에서 이미 도출된 마일스톤 범위 안의 업무를 CEO(대표 에이전트)와 CTO(기술 총괄 에이전트)가 보드 개입 없이 최대한 자율적으로 처리하도록 요청했다.

다만 그대로 두면 다음 모호함이 남는다.

- 어느 수준의 업무까지 두 역할이 바로 집행할 수 있는지
- 언제 보드용 이슈를 새로 만들어야 하는지
- 범위, 우선순위, 소유자를 누가 어디까지 바꿀 수 있는지
- 보고를 어떤 형식으로 남겨야 다음 실행자가 문맥을 복구할 수 있는지

이 모호함은 대기와 중복 보고를 만들고, 운영 오류가 생겼을 때 책임 경계를 흐린다.

## Decision

1. 이번 자율 운영은 "이미 승인된 마일스톤" 안의 실행 업무에만 적용한다.
2. CEO와 CTO는 기존 마일스톤 내부의 하위 이슈 생성, 실행, 검증, 상태 갱신을 보드 승인 없이 진행할 수 있다.
3. CEO는 범위 통제, 우선순위 해석, 운영 결함 추적, 보드 에스컬레이션(상위 의사결정 요청) 기준 확정의 최종 책임을 진다.
4. CTO는 기술 설계, 구현, 테스트, 기술적 하위 이슈 분해와 실행의 최종 책임을 진다.
5. 두 역할 모두 기존 승인 범위를 넘어서는 신규 마일스톤 추가, 기존 우선순위 재정렬, 파괴적 실행, 사람 또는 팀 간 최상위 소유권 변경은 보드 승인 없이 할 수 없다.
6. 자율 처리 불가 조건이 발생하면 즉시 보드용 이슈를 만들고, 막힌 이유와 필요한 결정 한 줄을 적는다.

## Autonomous Scope

자율 처리 가능:

- 기존 마일스톤 안에서의 하위 작업 분해
- 문서, 코드, 테스트, 검증 산출물 작성
- 같은 목표 안의 상태 전환과 진행 보고
- 운영 오류를 재현하고 원인 후보를 좁히는 조사
- 승인된 범위를 벗어나지 않는 권한 조정 제안서 작성

자율 처리 불가:

- 새로운 제품 범위 추가
- 기존 마일스톤의 목표 변경
- 우선순위를 다른 마일스톤보다 앞세우거나 뒤로 미루는 결정
- 보드 전용 파괴적 실행
- 예산, 인력, 외부 계약, 조직 구조 자체를 바꾸는 결정

## Escalation Rule

다음 중 하나라도 해당하면 자율 처리 중단 후 보드용 이슈를 만든다.

1. 기존 마일스톤 문구만으로는 범위 해석이 둘 이상 가능하다.
2. CEO와 CTO 중 누구도 필요한 권한, 접근, 승인 경로를 갖고 있지 않다.
3. 범위, 소유자, 우선순위 셋 중 하나라도 바꿔야 해결된다.
4. 파괴적 실행 또는 대외 영향이 있는 조치가 필요하다.
5. 운영 결함을 현재 실행 안에서 고칠 수 없고, 그냥 종료하면 같은 대기가 반복된다.

## Reporting Rule

자율 처리 종료 시에는 이슈 댓글에 아래 네 줄을 남긴다.

- 상태: 지금 무엇이 끝났는지 또는 무엇이 막혔는지
- 변경: 코드, 문서, 테스트, 운영 조치 중 실제 변경 사항
- 위험: 남은 불확실성 또는 보드 판단 필요 여부
- 다음 조치: 소유자와 행동, 시한 한 줄

## Why

- 자율성은 속도를 높이지만, 범위 경계가 없으면 오히려 재작업이 늘어난다.
- CEO와 CTO의 책임축을 분리해야 대기 없이 결정을 내릴 수 있다.
- 운영 결함도 제품 결함처럼 이슈와 결정 기록으로 남겨야 재발을 줄일 수 있다.

## Consequences

- 기존 마일스톤 내부 업무는 더 적은 보드 개입으로 진행한다.
- 대신 경계 밖 요구는 즉시 보드 이슈로 올려 감사 흔적을 남긴다.
- 이후 같은 유형의 자율 업무 요청은 이 문서와 [NIT-52](/NIT/issues/NIT-52)를 기준으로 해석한다.
